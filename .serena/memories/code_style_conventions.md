# コードスタイルと規約

## 基本原則

### CLAUDE.mdの指針に従う
- 日本語でのコミットメッセージ
- 推測よりも計測を重視
- 型安全性を最優先
- 最低限のテストで冗長を避ける

## TypeScript規約

### 命名規則
- **変数・関数**: camelCase
- **クラス・インターフェース**: PascalCase
- **定数**: UPPER_SNAKE_CASE
- **ファイル名**: kebab-case
- **コンポーネント**: kebab-case（例: `task-list`, `auth-button`）

### 用語統一
- **Todo**: Todoistに登録される作業項目の総称
- **TaskTodo**: `@task`ラベルが付いたTodo
- **GoalTodo**: `@goal`ラベルが付いたTodo

### 型安全性
- 型推論を積極的に活用
- `any`の使用は最小限に
- 厳密な型チェック（`strict: true`）
- インターフェースによる明確な契約定義

## Lit（フロントエンド）規約

### コンポーネント設計
- Reactive Controllerパターンの採用
- kebab-case命名のカスタム要素
- UIコンポーネントは`components/ui/`に配置
- 責任分離: ビジネスロジック（Controller）とUI（Component）

### ディレクトリ構造
```
frontend/src/
├── app-element.ts              # アプリケーション層
├── components/                 # コンポーネント層
│   ├── ui/                    # 再利用可能UIコンポーネント
│   └── *.ts                   # 機能パネルコンポーネント
├── controllers/               # コントローラー層
├── services/                  # サービス層
├── types/                     # 型定義
├── utils/                     # ユーティリティ
├── config/                    # 設定
└── styles/                    # スタイル
```

## Deno（バックエンド）規約

### フォーマット設定
- インデント: スペース2文字
- 行幅: 80文字
- セミコロン: 必須
- シングルクォート: false（ダブルクォート使用）

### ファイル構成
- `main.ts`: エントリーポイント
- `deno.json`: 設定ファイル
- 型安全性とDenoの標準ライブラリを活用

## Linting・Formatting

### フロントエンド（ESLint）
- `@typescript-eslint`プラグイン使用
- 明示的なアクセス修飾子必須
- 未使用変数エラー
- `any`型は警告
- 型定義は`interface`を優先

### Deno（標準ツール）
- `deno fmt`: コードの整形
- `deno lint`: コードのリント
- `deno check`: 型チェック

## Git規約

### コミットメッセージ
- 日本語で記述
- プレフィックス使用: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`
- 例: `feat: TaskTodo完了統計グラフを追加`

### ブランチ戦略
- `main`: メインブランチ
- 機能ブランチ: `feature/機能名`
- バグ修正: `fix/バグ名`

## テスト規約

### フロントエンド（Vitest）
- ファイル名: `*.test.ts`
- 最低限のテストで冗長を避ける
- 既存機能の破綻確認と新機能の動作確認

### テスト方針
- 変更点が多くなる予想箇所にはテストを書かない選択肢もある
- 明示的にテストを求められた場合は必ず記述

## パフォーマンス考慮

### フロントエンド
- 中規模タスク数（〜500件）を想定
- 全ページ取得によるフィルタ機能
- 大量タスク（1000件超）での対応が必要な場合は仮想化を検討

### API呼び出し
- 5分間のインメモリキャッシュ
- ラベルフィルタによる効率的API呼び出し
- タスク変更時の自動キャッシュクリア
