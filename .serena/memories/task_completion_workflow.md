# タスク完了時のワークフロー

## CLAUDE.mdに従った開発フロー

### 1. 修正案の計画
- 問題の分析と解決方針の決定
- アーキテクチャへの影響を検討
- 既存パターンとの整合性確認

### 2. 修正の実装
- 既存のライブラリ・モジュールの活用を優先
- インターネット上の既存解決法を調査
- コードベース内の参考実装を確認
- 自前実装は最後の手段

### 3. テストの作成（変更箇所が少ない場合）
- 最低限のテストで冗長を避ける
- 既存機能の破綻確認
- 新機能の動作確認

### 4. テストの実行と修正
- テストが通るまで修正を継続
- エラーの根本原因を追求

### 5. 品質チェック（必須）

#### フロントエンド
```bash
cd frontend
pnpm run type-check    # TypeScript型チェック
pnpm run lint          # ESLintによるコードチェック
pnpm run test          # Vitestでテスト実行
```

#### プロキシサーバー
```bash
cd proxy
deno task check        # 型チェック
deno task lint         # コードのリント
deno task fmt          # コードの整形
```

#### Cronサービス
```bash
cd cron
deno task check        # 型チェック
deno task lint         # コードのリント  
deno task fmt          # コードの整形
```

### 6. 既存テストの確認
- 全てのテストスイートが通ることを確認
- リグレッションの早期発見

### 7. コミット（明示的に依頼された場合のみ）
- 日本語でのコミットメッセージ
- 適切なプレフィックス使用
- 変更内容の簡潔な説明

## エラー対応方針

### 推測よりも計測
- エラーメッセージからの表面的修正は禁止
- 根本原因の追求と類似エラーの確認
- 検証方法の例：
  - Web検索による解決法調査
  - コードベース全体の関連コード読解
  - printデバッグによる動作確認

### 検証手法
1. **Web検索**: 同様の問題と解決法
2. **コード読解**: ライブラリ内部も含む関連コード
3. **printデバッグ**: 実際の動作を確認

## 品質保証の重要性

### 型安全性の確保
- TypeScript型チェックで型エラーを排除
- 実行時エラーの最小化
- インターフェースによる契約の明確化

### リンティングの実行
- コードスタイルの統一
- 潜在的なバグの発見
- ベストプラクティスの強制

### テストの実行
- 機能の正常動作確認
- 既存機能への影響確認
- リグレッションの防止

## ドキュメント更新

### CLAUDE.mdの更新確認
- 機能追加や変更時に必ず確認
- 古い内容の更新
- 新機能の説明追加

### メモリファイルの更新
- プロジェクト構造の変更
- 新しい開発パターンの追加
- 制約や注意点の記録
